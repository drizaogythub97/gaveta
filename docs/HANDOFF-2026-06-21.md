## 2. Revisão de segurança — cobertura e resultado

**Resultado: nenhuma vulnerabilidade HIGH ou MEDIUM.** Arquitetura coerente
com as regras inegociáveis do `CLAUDE.md`. Arquivos lidos e veredito:

| Superfície | Arquivo(s) | Veredito |
|---|---|---|
| Sessão / proteção de rotas | `lib/supabase/middleware.ts`, `proxy.ts`, `lib/supabase/server.ts` | `getUser()` em todo lugar; `getSession()` **não** aparece em código (só em comentários/docs). ✅ |
| Callback de auth | `app/auth/callback/route.ts` | `type` validado contra allowlist (`ALLOWED_TYPES`); `next` forçado a começar com `/` **e** sempre prefixado com `origin` → sem open redirect. ✅ |
| `service_role` | `app/(app)/minha-conta/actions.ts`, `lib/env.ts` | Uso único (`admin.auth.admin.deleteUser`) em Server Action; `getServiceRoleKey()` lê env **sem** `NEXT_PUBLIC_`. Nunca chega ao cliente. ✅ |
| RLS (todas as tabelas) | `0001`, `0003`, `0004` | RLS habilitado em `profiles`, `products`, `sales`, `sale_items`, `product_barcodes`, `preferences_fees`; políticas `auth.uid() = user_id`. ✅ |
| RPC `register_sale` | `0001`/`0004` | `security invoker` (roda sob RLS do usuário); guarda `auth.uid() is null`; valida método/parcelas; baixa de estoque restrita a `user_id = v_user`. ✅ |
| Storage `brand-logos` | `0005` | Policies insert/update/delete exigem `(storage.foldername(name))[1] = auth.uid()::text`; path sempre `${user.id}/<uuid>.<ext>`. ✅ |
| Upload de logo | `app/(app)/preferencias/actions.ts` | Valida tamanho (≤1,5 MB), MIME **e** magic bytes (PNG/JPEG/WebP); filename via `crypto.randomUUID()`. ✅ |
| Validação de entrada (servidor) | todas as `actions.ts` | Zod/validação manual no servidor; erros ao usuário genéricos (sem vazar SQL/stack). ✅ |
| XSS | `app/layout.tsx`, `lib/theme/cookie.ts` | Único `dangerouslySetInnerHTML` é `THEME_INIT_SCRIPT` — **constante estática**, sem interpolação de dado do usuário. ✅ |

### Observações de defesa-em-profundidade (NÃO são vulnerabilidades, não bloqueiam deploy)
1. **`register_sale` não valida que o `product_id` enviado pertence ao
   usuário** ao inserir em `sale_items`. Sem impacto real: o insert é coberto
   por RLS (`user_id = auth.uid()`) e a baixa de estoque filtra
   `user_id = v_user`. No máximo gravaria um `product_id` "órfão" na própria
   linha — sem vazamento (leitura é RLS-restrita) e sem afetar estoque alheio.
   *Melhoria opcional pós-MVP:* validar posse do `product_id` dentro da RPC.
2. **`profiles` não tem policy de INSERT/DELETE explícita.** Correto por
   design: insert vem do trigger `handle_new_user` (`security definer`),
   delete cascateia de `auth.users`. Sem ação.

   portabilidade Cloudflare (nenhuma API proprietária da Vercel é usada —

   `{{ .SiteURL }}/auth/callback?token_hash={{ .TokenHash }}&type=...&next=...`.

     (decisão consciente — são para colar no painel na véspera). O próximo

   `noreply@mail.app.supabase.io` e caem em spam. Supabase → Authentication →
   SMTP Settings.
3. **Site URL + Redirect URLs de produção** — hoje apontam para
   `http://localhost:3000`. Trocar para o domínio da Vercel + `/auth/callback`.
4. **Religar "Confirm email"** — foi desligado na Fase 3 para acelerar testes.
