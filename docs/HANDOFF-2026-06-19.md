### 2.2 Fase 4 — Produtos e Frente de Caixa
- CRUD de produtos em `/produtos`, `/produtos/novo`, `/produtos/[id]/editar`.
- Formulário compartilhado com **toggle "Controlar estoque? Sim/Não"** que
  esconde o campo de quantidade quando "Não" (sob demanda, ex.: marmita).
- Campo de preço com **máscara de moeda automática** (digita dígitos →
  exibe "R$ 10,50" → submete decimal "10.50") via novo componente
  `CurrencyInput` (`components/app/currency-input.tsx`).
- Frente de Caixa em `/caixa`:
  - Input único com **foco automático**; aceita **leitura por código de
    barras** via scanner USB (keyboard wedge: digita + Enter).
  - **Autocomplete por nome** com debounce 220ms.
  - **Item avulso** quando não bate com produto cadastrado (também com
    moeda mascarada no campo de valor).
  - Botões `+ / −` para quantidade, edição livre da quantidade, remoção.
  - Total grande, `aria-live` para SR.
  - Forma de pagamento: Dinheiro (default, com input de valor recebido +
    troco), Pix, Cartão de Débito, **Crédito à vista**, **Crédito parcelado**
    (com input de parcelas 2x–12x), Vale alimentação/refeição.
  - Cálculo de **taxa estimada** e **valor líquido** em tempo real usando as
    preferências de taxas do usuário.
  - Registro via RPC `register_sale(items, payment_method, installments, fee_amount)`.

### 2.3 Fase 5 — Dashboards
- **Painel `/dashboard`** com 4 KPIs em paralelo (Faturamento hoje,
  Faturamento mês, Vendas hoje, Estoque baixo ≤ 5) e 4 tiles de atalho.
- **Estoque `/estoque`** server-side filtra `track_stock=true`. Cada produto
  tem ação **inline "Atualizar quantidade"** (modo `set`) e **"Receber
  entrada"** (modo `add`) com preview do total final. Server action
  `updateStock` com RLS.
- **Financeiro `/financeiro`** com seletor de período via search params
  (Hoje / 7d / 30d / Mês / Personalizado), filtro por **forma de pagamento**
  (multi-select em pílulas), e card grande com **Bruto · Taxas · Líquido**.
  Cada venda mostra status, método, parcelas (quando aplicável), itens e
  ação reversível Estornar/Reativar (só troca `status`, não devolve estoque
  ainda).

A **assinatura atual** do RPC é:

## 7. Contexto técnico que NÃO PODE esquecer

### Stack
- **Next.js 16.2.9 (App Router, Turbopack)**, React 19.2.4.
- **Tailwind v4** (CSS-first via `@theme` em `app/globals.css`).
- **shadcn/ui preset `base-nova`** (usa `@base-ui/react`, **não** Radix).
  - Atenção: o `Button` daqui usa `@base-ui/react/button`, que **não**
    suporta `asChild`. Para "link com cara de botão" usar
    `buttonVariants(...)` no `className` do `<Link>` (vários exemplos no
    código).
- **Supabase** via `@supabase/ssr` 0.12 + `@supabase/supabase-js` 2.108.
- **Zod 4.4.3** — usa `z.email("msg")` (não `z.string().email()`) e
  `z.literal("on", { error: "msg" })` (não `errorMap`).
- **lucide-react 1.21** para ícones.
- **react-image-crop 11.0.10** para o crop do logo.
- Vitest 4.1 para unit/RLS; Playwright 1.61 para E2E (E2E ainda não está
  escrito — Fase 8).

### Next 16 / proxy
- `middleware.ts` foi renomeado para **`proxy.ts`** (Next 16). A função
  exportada chama-se `proxy(request)`. Está em `proxy.ts` na raiz do
  projeto e delega para `lib/supabase/middleware.ts`. **Não recriar
  `middleware.ts`.**

### Convenções do projeto (de `CLAUDE.md`)
- TypeScript estrito, sem `any` desnecessário.
- Server Components por padrão; `"use client"` só quando precisa.
- Nomes/commits em inglês; **textos de interface em pt-BR**.
- Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`, ...).
- Acessibilidade alta (contraste AA, alvos ≥44px, fontes grandes, labels
  aria) — os inputs/forms usam `h-14 text-lg` como padrão.
- **`service_role` nunca no cliente.** Único uso atual: em
  `app/(app)/minha-conta/actions.ts` para `admin.auth.admin.deleteUser`.
- **`supabase.auth.getUser()` sempre no servidor**, nunca `getSession()`.
- **Toda entrada validada por Zod no servidor.**
